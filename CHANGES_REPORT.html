Section 1

Executive Summary

Tuangou (微团购) is a full-stack Chinese group-buying e-commerce platform built on Next.js 15 (App Router), Supabase (PostgreSQL + Auth), and a Spring Boot backend. The platform serves a bilingual (English / Simplified Chinese) customer base and supports the complete retail lifecycle: product browsing, cart management, group-buy checkout, delivery and community pickup logistics, order tracking, and an administrative dashboard. Given the platform's transactional nature, correctness and reliability at the API and state management layers are prerequisites for both user trust and operational accuracy.

A comprehensive code audit was conducted across all areas of the frontend codebase — React providers, Next.js API route handlers, shared utility libraries, page components, and the internationalisation layer. Issues were identified through deep static analysis, examining data-flow correctness, error handling completeness, type safety, and UI/UX consistency. In total, 30 issues were identified and resolved across four parallel fix branches (fix/auth-profile, fix/cart-checkout, fix/order-api, fix/ui-ux), all of which have been merged into the development branch.

The changes delivered by this audit improve the platform across four key dimensions: stability (eliminating runtime crashes and silent error swallowing), security (preventing privilege escalation via DB errors, ensuring price integrity), correctness (fixing stale closures, mismatched constants, and type-unsafe casts), and user experience (replacing infinite loading states, broken images, and non-functional UI elements with clear, appropriate feedback). No feature additions were made — the scope is strictly repair and hardening of existing functionality.

Section 2

Audit Methodology

A deep static analysis was performed across all TypeScript source files in the repository, including React providers, Next.js route handlers, client-side page components, shared library utilities, and the internationalisation system. Each file was read in full, with attention paid to data-flow correctness, error propagation paths, React hook dependency arrays, TypeScript type safety, and consistency between related modules.

Severity Classification

Each identified issue was assigned one of four severity levels based on its potential impact:

  • Critical — Can cause data loss, incorrect order processing, runtime crashes, or security vulnerabilities under normal usage conditions.
  • High — Degrades user experience significantly or can lead to incorrect application state visible to the user.
  • Medium — Non-critical functional issues: non-functional UI elements, accessibility gaps, or locale-related display errors.
  • Low — Code quality, inconsistency, or minor polish issues that do not affect core functionality but reduce maintainability or professionalism.

Parallel Branch Strategy

To allow simultaneous implementation without merge conflicts, the 30 issues were grouped by domain into four non-overlapping fix branches. Each branch targeted a distinct set of source files, ensuring that no two branches modified the same file. The workflow for each branch was: identify and fix issues, commit with descriptive messages, push to remote, open a pull request for review, and merge into development. This approach allowed all four tracks to progress in parallel while preserving a clean, reviewable git history.

Scope Note Two files were included in the audit but required no changes: src/app/api/v1/me/route.ts (logic already correct) and src/components/storefront/ProductCard.tsx (already had correct alt text). These are documented in Section 5.
Section 3

Change Groups

3.1   Branch: fix/auth-profile  —  3 issues fixed

Files modified: src/providers/AuthProvider.tsx, src/app/api/_lib/auth.ts
Files audited (no change needed): src/app/api/v1/me/route.ts
#6 Auth profile load failure invisible to UI Critical
What was wrong When the /api/v1/me profile fetch timed out or threw an error (a 10-second race condition existed), the code silently fell back to a half-populated user object. No error state was exposed to the rest of the application. Users could proceed through the entire checkout flow with missing name, phone number, or delivery address data.
Why it mattered Silent failures in authentication mean corrupted sessions are indistinguishable from valid ones. Users placing orders without a fully loaded profile creates downstream fulfillment failures and customer support burden.
How it was fixed Added profileError: string | null to the AuthContextValue interface and as a React state variable. It is set to "Profile failed to load" on timeout or thrown error, cleared on a successful profile fetch, and cleared on sign-out. Any UI component can now read profileError and surface an appropriate warning.
#12 Profile query errors silently downgrade admins High
What was wrong In auth.ts, when the profile database query failed for any reason, the code logged a console.warn and defaulted to role: "user". A real database error — permission denied, connection timeout — would silently strip an admin of their privileges with no indication of what occurred.
Why it mattered An administrator could lose access to the admin panel mid-session due to a transient infrastructure issue, causing operational disruption and confusion.
How it was fixed The code now distinguishes between PostgREST error code PGRST116 ("no row found" — expected for brand-new users without a profile row yet; safe to default to "user") and all other error codes (unexpected — now throws, causing the request to fail with a 500 rather than proceeding with incorrect permissions). A comment documents this distinction for future maintainers.
#24 Unsafe as any cast on WeChat provider type Low
What was wrong A provider: 'wechat' as any cast was silencing a TypeScript type error. If Supabase's OAuth provider type definition changed in a future SDK update, the code would still compile but fail silently at runtime.
How it was fixed Replaced with "wechat" as OAuthProvider, where OAuthProvider is derived from Parameters<typeof supabase.auth.signInWithOAuth>[0]["provider"] — a type-safe alias that will surface any future SDK incompatibility at compile time rather than at runtime.

3.2   Branch: fix/cart-checkout  —  7 issues fixed

Files modified: src/providers/CartProvider.tsx, src/app/(storefront)/cart/page.tsx, src/app/(storefront)/checkout/page.tsx, src/app/api/_lib/cart-helpers.ts, src/app/api/v1/cart/merge/route.ts, src/i18n/strings.ts
#1 CartProvider stale closure / circular dependency Critical
What was wrong The setQuantity useCallback captured a stale snapshot of the remove callback due to a circular dependency in their respective useCallback dependency arrays. This could cause cart quantity updates to operate on outdated state, resulting in incorrect quantities and data inconsistencies.
Why it mattered Cart state is the most critical user-facing data in the checkout flow. Stale closures here can lead to orders being placed with wrong quantities — a direct fulfillment and revenue integrity issue.
How it was fixed Refactored setQuantity to inline the remove logic directly, eliminating the circular dependency entirely. Both callbacks now declare only their true, direct dependencies.
#2 Delivery threshold constant never used Critical
What was wrong A DELIVERY_THRESHOLD_CENTS constant (= 15000, representing $150) was defined at the top of the checkout page, but the actual delivery qualification check used a hardcoded literal 5000 (= $50). The constant and the business logic were entirely inconsistent with each other.
Why it mattered Users could qualify for delivery at a threshold different from what any documentation or UI messaging implied, causing trust and fulfillment issues.
How it was fixed Replaced the hardcoded 5000 literal with the DELIVERY_THRESHOLD_CENTS constant so the defined value is used consistently throughout the checkout page.
#5 Cart merge race condition swallows all errors Critical
What was wrong In cart-helpers.ts, when a concurrent insert caused a unique constraint violation, the error was caught and treated as safe to ignore. However, the code did not verify the error was specifically a PostgreSQL 23505 (unique violation) — any error code was silently swallowed.
Why it mattered A real failure such as a permission error or connection drop would be treated as a benign race condition, causing silent cart data loss with no feedback to the user or caller.
How it was fixed Added an explicit check error.code === "23505" before treating the conflict as safe to ignore. All other error codes are now re-thrown. A comment documents the 23505 = unique_violation distinction for future maintainers.
#10 Cart merge silently drops invalid items High
What was wrong During guest-to-user cart merge, items with a missing productId or a quantity less than 1 were silently skipped. Users would lose cart items with no indication that anything had gone wrong.
How it was fixed Invalid items are now collected into a skippedItems array and included in the API response as { merged: [...], skipped: [...] }. The client layer can inspect the response and surface an appropriate warning to the user when items are skipped.
#11 Whitespace-only addresses pass checkout validation High
What was wrong Address field validation checked !address.line1, but a string consisting entirely of spaces (e.g. "   ") would pass this check, allowing orders with effectively blank delivery addresses to be submitted.
How it was fixed All address field checks updated to use .trim(): !address.line1.trim(), !address.city.trim(), and so on across all required address fields.
#14 Promo code input is completely non-functional Medium
What was wrong A promo code input field and "Apply" button existed in the cart page but were wired to do nothing — state was set on input but was never validated against a backend or applied to the cart total.
Why it mattered Users attempting to enter promo codes received no feedback and could not understand why their code was not working, creating frustration and support requests.
How it was fixed The promo code input, Apply button, and associated state were removed entirely. A non-functional placeholder feature with no backend integration is worse than the absence of the feature.
#20 Checkout delivery/pickup mode switch gives no feedback Medium
What was wrong When the cart total crossed the delivery qualification threshold and the checkout mode silently switched between delivery and pickup, the community pickup dropdown would reset with no explanation to the user.
How it was fixed Added a bilingual contextual info banner: "Your order qualifies for home delivery!" when above threshold, and "Add more items to qualify for home delivery (min. $150)" when below. New i18n keys were added to strings.ts in both English and Chinese.
#21 Hardcoded "Free / 免费" in checkout and cart Low
What was wrong The delivery fee display used a hardcoded bilingual string literal "Free / 免费" rather than routing through the application's internationalisation system.
How it was fixed Added a checkout.deliveryFree key to strings.ts with en: "Free" and zh: "免费". Both the checkout and cart pages now use t("checkout.deliveryFree").

3.3   Branch: fix/order-api  —  6 issues fixed

Files modified: src/app/api/v1/orders/route.ts, src/components/storefront/OrderDetailCard.tsx, src/app/(storefront)/account/orders/page.tsx, src/lib/repository.ts
#3 Product force-cast crashes order route Critical
What was wrong productMap.get(item.productId)! used a non-null assertion operator. If a product was deleted between when the user loaded their cart and when they submitted checkout, this line would throw an unhandled runtime exception, returning a bare 500 with no descriptive error message.
Why it mattered A deleted product in a user's cart would crash the order creation endpoint entirely. Users would receive a generic server error with no indication of what went wrong or how to resolve it.
How it was fixed Removed the ! assertion. Added an explicit null guard: if !product, return apiError(409, "One or more products are no longer available"). This gives the client a structured, actionable error response.
#4 Order route does not validate price types Critical
What was wrong price_cents values retrieved from the database were used directly in order total calculations without validating that they were positive integers. Floating-point prices would silently truncate; zero or negative prices were not rejected.
How it was fixed Added validation immediately after the product existence check: if (!Number.isInteger(product.price_cents) || product.price_cents <= 0) return apiError(400, "Invalid product price data"). This ensures that any data integrity issue in the products table surfaces as a structured API error rather than producing a corrupt order total.
#9 Orders list silently returns empty on API failure High
What was wrong The listOrders() function in repository.ts caught all errors and returned an empty array []. When the backend was unreachable, users would see a completely empty order history page with no error indication — they might believe their past orders had been lost.
How it was fixed Removed the catch-and-swallow pattern from listOrders(); the function now re-throws. The calling page (account/orders/page.tsx) was updated to catch the error and display a fetchError state with the message "Could not load orders. Please try again."
#19 Duplicate React keys in order detail list Medium
What was wrong The key prop in the order items list used the pattern ${"{order.id}"}-${"{it.productId}"}. If the same product appeared more than once in an order, this would generate duplicate React keys, breaking React's reconciliation algorithm and potentially causing visual glitches.
How it was fixed Updated to ${"{order.id}"}-${"{it.productId}"}-${"{index}"}, using the array index as a guaranteed uniqueness tiebreaker within each order's item list.
#23 Activity logging has no retry on failure Low
What was wrong logActivity() fired an API call with no retry logic. If the backend was temporarily unavailable, analytics events were permanently and silently lost.
How it was fixed Added storePendingActivity() and flushPendingActivities() helpers. On failure, events are persisted in localStorage under the key tuangou.pendingActivities (capped at 10 entries; oldest events are discarded when the cap is exceeded). On each subsequent logActivity() call, any pending events are flushed first before the new event is sent.
#25 / #26 Minor order route cleanup Low
Issue #25 Added a dedicated fetchError state variable to the orders page, wired to the re-thrown error from listOrders() (connected to the #9 fix above).
Issue #26 Removed the redundant capitalised Idempotency-Key header lookup from the order route. Next.js normalises all incoming headers to lowercase, so only the lowercase idempotency-key lookup is needed.

3.4   Branch: fix/ui-ux  —  6 issues fixed

Files modified: src/app/(storefront)/product/[slug]/page.tsx, src/app/(storefront)/account/page.tsx, src/components/common/Navbar.tsx, src/lib/utils.ts, src/i18n/strings.ts, src/i18n/LanguageProvider.tsx, src/app/admin/page.tsx
Files audited (no change needed): src/components/storefront/ProductCard.tsx
#7 Product page infinite loading on unknown slug High
What was wrong If a product slug didn't match any existing product — for example, a deleted product or a mistyped URL — the page would display a loading spinner indefinitely with no 404 response or redirect to a meaningful page.
Why it mattered Users landing on a bad product URL would be stuck with no way to navigate back except using the browser's back button, creating a poor and confusing experience.
How it was fixed Added a productsLoaded state flag. Once products have finished loading and the slug still doesn't match any product, a centered "Product not found" card is rendered with a navigation link back to /shop. New i18n keys product.notFound and product.notFoundBody were added to strings.ts in both English and Chinese.
#8 No avatar image fallback on account page High
What was wrong The user avatar <img> element on the account page had no onError handler. Expired or broken avatar URLs — common with OAuth providers where tokens have lifetimes — would display a broken image icon.
How it was fixed Added an avatarError React state and an onError handler on the image element. On load failure, the component renders a fallback initials div styled with bg-[#D94F2B] text-white — consistent with the existing avatar fallback in the Navbar component.
#13 HTML lang attribute never set on first load Medium
What was wrong document.documentElement.lang was only updated when the user actively changed the locale setting, never on initial page mount. Every first page load had no lang attribute on the <html> element.
Why it mattered Search engines and screen readers rely on the lang attribute to identify the page language. This was an SEO and accessibility gap affecting every single page load for all users.
How it was fixed The LanguageProvider initialization effect now sets document.documentElement.lang immediately on mount (to "zh-CN" or "en" as appropriate) in addition to updating it on subsequent locale changes.
#17 Product gallery shows the same image 4 times Medium
What was wrong The product detail page had a 4-slot thumbnail strip that rendered the same product.imageUrl in all 4 thumbnail positions. This was unfinished placeholder code for a multi-image feature that was never completed.
Why it mattered Users would see 4 identical thumbnails and try clicking them expecting different views, damaging product page credibility and raising questions about the platform's quality.
How it was fixed The thumbnail strip was removed entirely. The page now renders a single clean main product image without the incomplete multi-image gallery UI.
#18 / #29 formatPrice always outputs USD regardless of locale Medium
What was wrong The formatPrice(cents, locale?) utility in utils.ts accepted a locale parameter but always returned a USD-formatted string ($XX.XX) regardless of what was passed. For a Chinese-market platform, displaying USD prices is functionally incorrect.
How it was fixed Implemented proper locale-aware formatting using Intl.NumberFormat: when locale is 'zh' or 'zh-CN', the function outputs CNY format (¥XX.XX); all other locales continue to receive USD format ($XX.XX). The change is fully backward-compatible — callers that don't pass a locale continue to receive USD output.
#22 Navbar avatar initial doesn't match account page logic Low
What was wrong The Navbar derived the avatar initial using user.name.charAt(0), while the account page used a different derivation that fell back through multiple user object fields. This caused inconsistent display of the user's initial depending on which part of the UI was rendering it.
How it was fixed Extracted a shared getDisplayName(user): string utility function into utils.ts with a canonical fallback chain: fullName → name → email prefix → "User". Both the Navbar and the account page now call this shared function, ensuring the displayed name and initial are always consistent.
#30 No loading/error state on admin stats dashboard Low
What was wrong While admin dashboard statistics were loading, all stat cards displayed blank or zero values with no loading indicator. If the stats fetch failed, nothing in the UI indicated an error had occurred.
How it was fixed Added skeleton loading cards using animate-pulse bg-gray-200 that display while stats are null. A bg-red-50 text-red-700 border border-red-200 error banner is rendered above the skeleton cards if the fetch fails, providing clear feedback to administrators.
